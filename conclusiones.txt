Conclusiones del ejercicio

1. La automatizacion E2E valida de extremo a extremo el flujo solicitado:
- Agregar dos productos al carrito.
- Visualizar el carrito.
- Completar el Guest Checkout.
- Finalizar hasta el mensaje "Your order has been placed!".

2. Hallazgos tecnicos relevantes
- El checkout de OpenCart carga sus formularios y botones por AJAX, por lo que la estabilidad depende de esperas explicitas sobre paneles y botones de continuacion.
- El sitio usa HTTPS en el checkout con un certificado no confiable para herramientas simples como PowerShell; en navegador automatizado se resolvio aceptando certificados inseguros.
- Los productos seleccionados aparecen con marca "***" en el carrito, asociada a una advertencia de disponibilidad o cantidad. Aun asi, el flujo permite completar la compra y llegar a la confirmacion final.

3. Decisiones de implementacion
- Se utilizo Serenity BDD con Cucumber, Screenplay y Maven.
- La suite se organizo con la estructura clasica de runner, stepdefinitions, features, constants, ui y tasks para que la ejecucion pueda iniciarse directamente desde un runner.
- Se dejo la ejecucion principal en Failsafe para evitar la duplicacion de corridas con Surefire.
- Los datos de prueba y mensajes esperados quedaron centralizados en clases de constants para hacer mas visible la parametrizacion del flujo.
- El runner clasico solicitado funciona correctamente con Serenity 5.3.7, aunque el framework ya lo marca como API en transicion; si a futuro se actualiza la estrategia de ejecucion, conviene revisar la alternativa recomendada por Serenity.

4. Evidencia de validacion
- La ejecucion verificada localmente fue: mvn clean verify
- El reporte de Serenity se genera en: target/site/serenity/index.html

5. Recomendaciones
- Si el sitio cambia textos o estructura del checkout, revisar primero los localizadores de los paneles AJAX y del mensaje final.
- Mantener Java, Maven, Chrome y las dependencias de Serenity actualizadas para reducir problemas de compatibilidad entre navegador y WebDriver.
